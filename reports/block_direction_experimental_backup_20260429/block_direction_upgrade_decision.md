# Block Direction Upgrade Decision

## Decision

`BlockDirectionOptimizerV4Fast` should be the main block-direction branch moving forward.

`BlockDirectionOptimizerV3` should be kept as an experimental stress-specialist reference branch, not as the main line.

The reason is straightforward:

- V4 keeps the core block-direction novelty.
- V4 is much faster than V3.
- V4 is much better than V3 on standard neural tasks.
- V4 preserves the main stress-task strengths that made the block branch interesting.
- V4 still does not beat the strongest conventional optimizers broadly enough to be called a new default optimizer.

## What V4 Keeps

V4 keeps the actual novel principle of the block branch:

- parameters are split into blocks
- each block chooses among candidate directions
- direction choice is trust-scored rather than generated by Adam moments
- trusted direction memory still exists
- structured matrix consensus still exists
- stable consensus still exists

The core V4 candidate set is:

- `gradient`
- `stable_consensus`
- `trusted_direction`
- `matrix_consensus`

For 2D matrices, consensus is built from row and column normalized gradient structure.

For convolution/filter tensors, V4 now reshapes weights to `[out_channels, -1]` and applies the same row/column consensus at filter level. This is a real structural change, not a rename.

## What V4 Removes Relative To V3

V4 removes or de-emphasizes the parts of V3 that were expensive and not paying for themselves:

- no wide default candidate menu
- no per-candidate recoverability inside the hot path
- no default projection or orthogonal escape path
- no stress-specialist default behavior
- no heavy blending default

These rules still matter as research ideas, but they should not define the default optimizer path because they were making the branch too slow and too brittle.

## Core Math

For each block `B_i`, V4 builds a small candidate set `D_i`.

The stable consensus direction is

`d_i^(cons) = normalize(d_i^(grad) + a_i d_i^(trust) + b_i d_i^(smooth) + c_i d_i^(matrix))`

where each coefficient only contributes when that component aligns positively with the current descent direction.

The matrix consensus direction is built on a matrix or filter view `G_i`:

- normalize row directions
- normalize column directions
- weight them by row/column energy
- combine them with a row/column mix

So the matrix consensus is not a vague "low-rank idea". It is a structured row/column directional field synthesis.

The trust score is

`T_i(d) = w_d A_i(d) + w_m M_i(d) + w_q Q_i(d) + w_s S_i(d) + w_c C_i(d) - w_o O_i(d) - w_f F_i(d) - w_k cost(d)`

where the positive terms reward:

- descent alignment
- memory coherence
- improvement history
- norm stability
- consensus support

and the negative terms penalize:

- oscillation
- conflict
- candidate cost

Selection is winner-take-all by default:

`d_i^* = argmax_d T_i(d)`

Step magnitude is block-energy normalized, not Adam-style directional preconditioning:

`alpha_i = eta * lambda_i * ||g_i|| / (sqrt(E_i) + eps)^p`

with a parameter-relative cap to avoid destructive jumps.

## What V4 Gained

- much better standard-task behavior
- much lower runtime
- much stronger low-rank and direction-reversal behavior
- cleaner default math
- more defensible default rule set

## What V4 Lost

V4 lost **capability breadth**, not actual usefulness:

- fewer candidate types
- less built-in stress-specialist machinery
- recoverability removed from the hot path
- less exploratory behavior by default

That is a good tradeoff so far. V3 remains useful as a research sandbox, but not as the main optimizer.

Interpretation:

- V4 is usable on `digits_mlp`.
- V4 is very competitive on small dense neural tasks.
- V4 is still weak on `digits_cnn`.

So the filter-wise consensus extension helped make the branch structurally correct for conv filters, but it did **not** yet make it a strong CNN optimizer.

## PINN Probe

Focused PINN probe results:

| Task | V4 best loss | Best baseline |
| --- | ---: | --- |
| `pinn_harmonic_oscillator` | 0.3891 | `adamw_lbfgs_hybrid` at 0.0129 |
| `pinn_poisson_1d` | 0.0246 | `adamw_lbfgs_hybrid` at 1.3e-09 |
| `pinn_heat_equation` | 0.00889 | `lbfgs` at 6.0e-07 |

Interpretation:

- V4 beats V3 clearly on PINNs.
- V4 beats simple first-order baselines like `sgd_momentum` and sometimes `rmsprop`.
- V4 is nowhere near `LBFGS` or `AdamW -> LBFGS hybrid`.

So block direction selection is **not** currently a publishable PINN optimizer story.

## Runtime

Mean runtime per step:

- V3 mixed suite: `40.46 ms`
- V4 mixed suite: `14.12 ms`
- V3 CNN probe: `75.47 ms`
- V4 CNN probe: `26.42 ms`

Strong baselines are still much faster:

- `AdamW`: about `1.8 ms` in the CNN probe
- `RMSProp`: about `1.7 ms`
- `SGD+momentum`: about `1.6 ms`

So V4 is much more usable than V3, but still not cheap enough to claim broad practical superiority.

## Which Rules Actually Helped

Current evidence says keep:

- stable consensus
- matrix consensus
- lightweight trusted-direction memory
- winner-take-all selection
- block-energy normalization

Current evidence says do **not** make central:

- default recoverability
- projection-heavy default paths
- orthogonal escape default paths
- large candidate menus

## ARC Status

There is **no credible local ARC benchmark** in the current optimizer harness.

That means:

- there is no honest ARC performance claim yet
- there is no evidence that V4 is good or bad on ARC-style tasks
- any ARC claim right now would be speculative

If ARC matters, it needs a real benchmark branch with actual ARC-style supervised or program-synthesis tasks, not a proxy disguised as ARC.

## Final Verdict

### Should V3 or V4 be upgraded?

Upgrade **V4**.

Keep **V3** only as:

- a stress-specialist reference
- an experimental branch for extra candidate rules
- a place to test ideas before deciding whether they belong in the fast branch

### Is V4 publishable now?

Not yet as a broad optimizer.

It is closer to publishable as a **specialist structured direction-selection optimizer** if it can do one more thing:

- become clearly competitive on a wider standard neural family without losing its stress-task edge

### What still blocks a strong claim?

1. It still does not beat `RMSProp` or `SGD+momentum` broadly on standard neural tasks.
2. It is still weak on CNNs.
3. It is not a serious PINN contender against `LBFGS` and hybrid closure methods.
4. It has no credible ARC benchmark yet.

## Exact Next Step

The next upgrade should be **V4.1**, not a return to V3.

V4.1 should focus on the actual weak point:

- improve CNN/filter behavior without reintroducing V3 complexity

Most likely targets:

1. filter-consensus warmup or delayed trust activation for conv layers
2. channel-group consensus instead of only output-filter row consensus
3. separate step-scale policy for conv blocks vs dense blocks
4. digits_cnn-focused fair tuning against `AdamW`, `RMSProp`, and `SGD+momentum`

If V4.1 cannot materially improve CNNs while preserving the current wins, then the branch should be presented as:

- useful
- novel-ish at prototype level
- specialist for direction-stress and grouped-parameter settings

and **not** as a general replacement for strong conventional optimizers.
